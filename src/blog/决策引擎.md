
# 决策引擎

> 随着AI从“工具”走向“决策中枢”, 决策系统需要把数据、算法与业务动作串联成闭环的运营系统

## 产品定位

决策引擎是一个**通用决策平台**，将业务判断逻辑从代码中抽离，通过可视化配置、流程编排、统一调度规则与模型，实现可运营、可迭代、可量化的标准化决策能力。

目标用户：
- **业务运营**：自主配置和调整策略，无需开发介入
- **风控策略**：全生命周期管理决策，快速迭代规则与模型
- **开发人员**：通过自定义节点扩展引擎能力，对接业务系统

产品边界：
- 不是业务系统，不处理具体业务逻辑
- 不是通用工作流引擎，专注于决策判定而非流程流转
- 不是数据平台，依赖外部数据源而非自有存储，需要通过自定义函数来访问外部数据存储

## 背景

在业务的高速发展中，为了快速去拥抱变化，基本都是通过硬编码的方式将风控逻辑部署在业务逻辑中实现，但随着业务领域扩大，运营复杂度的提升，各种合规需求的适配，导致硬编码方式出现了策略分散无法管理、逻辑同业务强耦合、策略更新迭代率受限于开发、对接成本高等多种问题。

想象一下下列场景:
1. 在营销场景中, 业务改一条风控阈值、调整优惠券发放条件，需求设计, 开发编码, 测试验收, 发版上线，周期数天甚至几周。
2. 为了应对紧急的业务安全，技术需要快速上线防御垫片逻辑，长此以往相似判断逻辑散落在前端、网关、微服务、定时任务，多份代码副本，修改极易漏改，出现判定不一致、线上漏洞。
3. 模型在线上推理时突然遇到了大量关于误封的客诉, 需要紧急关闭模型，或者在某个特定条件下启用或者禁用模型。

在多年的安全研发和业务运营中，我们发现随着业务的发展，技术的迭代，运营规则和业务安全之间的界限变得越来越模糊。
此时，我们需要能够将业务流程中的**流程、规则、数据、模型**等全链路中绑定在代码里、难以变更、无法量化的业务决策能力，转化为业务可自主运营、快速迭代、可量化评估、可沉淀复用的标准化中台能力，解决业务策略频繁变更、规则散落在代码、策略效果无法量化等场景痛点，最终实现**业务提效、风险可控、成本下降、协调创新**四大核心收益。

## 核心设计

要为多种场景构建一个**通用的决策引擎**，关键在于设计一套 **“核心稳定、场景可插拔”** 的架构。可以**定义统一的“决策语言”**（元模型），构建**分层、模块化**的技术架构，并通过**流程、规则、数据、模型的差异化配置**来适配各个场景。

为了同时兼顾决策灵活性和执行引擎的高性能，我们选择基于rust开发的 [zen-engine](https://github.com/gorules/zen) 上去扩展开发执行引擎，其中包含了高度优化的 zen 表达式，以及 Json Decision Model。在此基础上，我们实现了通用的自定义节点函数规范 [zen-rule](https://github.com/republicroad/zen-rule)。


它的实现思路可以概括为：
- 表达式判断
- 流程编排
- 实时计算引擎
- 动态配置

决策引擎特点

- 决策流程是公开透明的，支持流程自动化
- 支持**规则热部署**，实现不停机更新
- 决策引擎提供自定义节点，其中提供了可插拔、易扩展自定义算子，可以模块化处理不同的场景问题
- 通过自定义节点去支持名单匹配，可以将沉淀的业务画像作用于决策过程
- 通过自定义节点支持模型。用**规则引擎**确保合规、底线等确定性逻辑；用**机器学习模型**处理非结构化数据、识别长尾风险等不确定性任务，两者相辅相成
- 以流程编排为核心，编排不同的节点去刻画不同的业务流程
- 引入自定义节点设计可插拔的拓展体系
- 通过**数据、模型、流程的差异化配置**来适配各个场景

### 任意流程

决策流由不同类型的节点编排而成，每个节点承载特定的决策功能：

- **开始/结束节点**：定义流程的入口及其参数schema和出口
- **表达式节点**：执行一条或多条规则（zen-expression 判定）
- **决策表节点**：基于决策表的多条件匹配，适用于复杂多条件决策场景
- **条件分支节点**：根据前序节点输出结果分流（Yes/No 或多路分支），同时也支持并行网关逻辑。
- **自定义节点**：通过 zen-rule 规范注册的扩展节点（名单匹配、滑动窗口、CEP 等）
	- **模型节点**：调用机器学习模型进行推理评分
	- **滑动窗口计数节点**: 
	- **名单节点**
	- HTTP 节点
	- 手机号节点
	- email节点
	- ip节点

编排引擎支持串行、并行、迭代器循环、子流程(WIP)等执行模式，满足从简单判定到复杂决策树的各种场景。

### 任意维度

决策引擎通过 **zen-expression** 支持从任意维度提取和计算特征：

- **原始字段**：直接使用入参中的字段（用户 ID、IP、设备指纹、金额等）
- **派生字段**：通过表达式计算产生新字段（如 `amount / 100`、`upper(city)`）
- **时序聚合**：滑动窗口计数/去重计数、时序数据查询聚合
- ~~**外部数据**：通过自定义节点从外部数据源实时获取（Redis 计数/去重、DB 查询、API 调用、名单匹配）
~~
表达式引擎支持字符串、数字、布尔、日期时间、数组、Context 等多种数据类型，以及丰富的内置函数（聚合、过滤、映射、条件判断）。

### 可插拔

基于 [zen-rule](https://github.com/republicroad/zen-rule) 规范实现插件化扩展：

- **自定义节点规范**：定义统一的节点接口（输入 schema、输出 schema、执行逻辑），注册后可在决策流中直接拖拽使用
- **内置算子**：字符串处理、数学计算、名单匹配、滑动窗口、CEP 事件序列等
- **模型扩展**：支持 ONNX、PMML 等格式的模型部署，通过自定义节点桥接
- **数据源扩展**：对接 Redis、MySQL、Postgresql、外部 HTTP API 等数据源

新节点开发完成后，通过插件注册机制加载即可生效。

### 决策编辑器

可视化的决策流编排工具，覆盖策略全生命周期：

- **拖拽画布**：所见即所得的流程编排，节点拖拽、连线条件配置
- **实时调试**：单条测试数据模拟执行，查看每步的输入/输出和路径
- **仿真回测**：用历史数据批量模拟策略效果，对比命中率、通过率变化
- **一键发布**：编辑完成经审批后，热部署到生产环境，秒级生效
- **版本管理(WIP)**：每次发布自动生成版本快照，支持回滚与历史对比

## 系统架构

决策引擎采用分层模块化架构，各层职责清晰、可独立扩展：

```
┌─────────────────────────────────────────────────────┐
│                    接入层                            │
│  HTTP API / 消息队列（Kafka/RocketMQ）   │
├─────────────────────────────────────────────────────┤
│                    编排层                            │
│  决策流解析、节点调度、条件路由、上下文传递            │
├─────────────────────────────────────────────────────┤
│                    执行层                            │
│  ┌─────────┐ ┌──────────┐ ┌──────────────────┐     │
│  │规则引擎  │ │模型引擎   │ │自定义节点引擎      │     │
│  │zen-expr │ │ONNX/PMML │ │名单/窗口/CEP/外部 │     │
│  └─────────┘ └──────────┘ └──────────────────┘     │
├─────────────────────────────────────────────────────┤
│                    数据层                            │
│  Redis / DuckDB / MySQL / 外部 API / OSS             │
├─────────────────────────────────────────────────────┤
│                    存储层                            │
│  决策流配置、规则定义、模型文件、审计日志、监控指标    │
└─────────────────────────────────────────────────────┘
```

**各层职责：**

- **接入层**：提供多协议接入能力（SDK、HTTP、gRPC、消息队列），统一请求格式，负责鉴权与限流
- **编排层**：解析决策流定义，按 DAG 图调度节点执行顺序，处理条件分支与并行逻辑，维护上下文传递
- **执行层**：核心执行单元，规则引擎基于 zen-engine（Rust）实现高性能表达式求值；模型引擎负责模型加载与推理；自定义节点引擎执行插件化算子
- **数据层**：封装对外部数据源的访问（实时计数、名单查询、特征获取），提供统一的数据抽象接口
- **存储层**：持久化决策流配置、规则定义、模型文件、审计日志、监控指标等

**部署架构：**

- 执行引擎无状态，支持水平扩展
- 编排层与执行层可分离部署，编排层负责调度，执行层专注计算
- 配置变更通过配置中心推送，执行引擎热加载无需重启

## 规则引擎

规则引擎是决策的核心执行单元，基于 [zen-engine](https://github.com/gorules/zen)（Rust）和 [zen-expression](https://github.com/republicroad/zen-rule) 实现：

- **zen-expression**：业务优先的表达式语言，支持字符串、数字、布尔、日期时间、数组等数据类型，提供丰富的内置函数（字符串处理、集合操作、数学运算、日期计算等）
- **决策表**：多条件多输出的表格化规则定义，支持区间匹配、枚举匹配、优先级排序
- **评分卡**：基于打分累加的模式，适用于信用评分、风险评级等场景
- **规则集**：多条规则的集合，按优先级顺序执行，支持命中即返回或全部执行后聚合
- **冲突解决**：明确优先级策略（指定优先级/最先匹配/最精确匹配），避免规则冲突

表达式语法示例：

```c
// 条件判断
user.age >= 18 and user.country in ['CN', 'HK']

// 集合操作
some(order.items, #.price > 1000)

// 聚合计算
sum(transaction.amount) > 50000
```


## 风控引擎

> 风控的本质就是对可能发生风险的任意业务维度进行访问控制。

决策引擎是风控的**大脑**，覆盖风控全生命周期：

- **事前**：通过规则和模型做准入判断（注册审核、交易拦截），从源头控制风险
- **事中**：实时监控业务行为，滑动窗口计数、CEP 事件序列识别异常模式（短时间内高频操作、异地登录等）
- **事后**：离线复盘与策略优化，仿真回测新策略效果，持续调优规则阈值

基于场景输入的数据，使用表达式和函数进行处理得到派生维度，再通过决策流编排进行复杂的逻辑运算，最终输出决策结果。可视化操作支持判断逻辑的多种组合与流程链路配置，支持快速部署上线，使得策略迭代速度能够从周级缩短到分钟级。这种模式帮助风控策略人员全生命周期管理决策，实现高效能的规则管理、灵活的流程配置、决策信息沉淀，持续提高风控质量。

## 算法

决策引擎支持将机器学习模型作为决策流中的一个节点，与规则互补：

**模型部署方式：**
- **ONNX Runtime**：直接加载 ONNX 格式模型，本地高性能推理
- **PMML**：加载 PMML 格式模型，适用于标准化的评分卡/回归/树模型
- **API 代理**：将外部模型服务（TensorFlow Serving、Seldon、自定义服务）封装为模型节点
- **模型版本管理**：多版本共存，支持 AB 测试和灰度发布

**特征工程：**
- 原始字段通过 zen-expression 进行特征变换
- 滑动窗口算子提供时序特征（频率、趋势、周期）
- 外部数据源算子补充画像特征（历史行为、关联数据）

**规则与模型的协同：**
- 规则引擎处理确定性逻辑（合规检查、底线拦截），保障可解释性
- 机器学习模型处理非确定性任务（异常检测、欺诈评分），处理长尾风险
- 两者在决策流中可任意组合编排：先用规则快速过滤，再对可疑请求调用模型深度评分

典型模式：规则前置过滤 → 模型评分 → 规则后置兜底 → 决策输出
